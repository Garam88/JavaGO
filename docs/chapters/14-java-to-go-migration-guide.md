# 14. Java -> Go 마이그레이션 가이드(선택)

## 서비스 분리 전략

마이그레이션의 목표는 "언어 전환"이 아니라 "위험을 통제한 성능/운영 개선"입니다.  
전면 재작성(big-bang)보다, 트래픽 경계가 명확한 기능부터 점진 분리하는 전략이 안전합니다.

### 어떤 서비스를 먼저 분리할까

우선순위 기준:

1. CPU/메모리 사용량이 높아 효율 개선 여지가 큰 기능
2. 도메인 경계가 명확해 독립 배포가 쉬운 기능
3. 변경 빈도가 높아 배포 속도 이점이 큰 기능
4. 외부 의존이 상대적으로 단순한 기능

예: `go-commerce-api`에서 "주문 조회/캐시 고도화 API"처럼 읽기 부하가 큰 경로를 먼저 분리하는 방식.

### 분리 방식

1. 기능 단위 추출(추천)
2. 트래픽 샤딩(일부 사용자/테넌트만 Go 경로)
3. 백그라운드 워커부터 분리(비동기 경로)

초기에는 동기 API 핵심 경로보다 비동기 소비자(이벤트 핸들러)부터 옮기면 위험이 낮은 경우가 많습니다.

### 마이그레이션 단계 플레이북

1. 대상 기능 SLA/지표 정의
2. API/이벤트 계약 고정(OpenAPI/Proto/JSON Schema)
3. Go 구현 + shadow traffic 검증
4. canary 배포(예: 1% -> 10% -> 50% -> 100%)
5. rollback 기준 충족 시 즉시 복귀

핵심은 "항상 되돌릴 수 있는 상태"를 유지하는 것입니다.

## 공통 라이브러리/SDK 설계

Java와 Go를 함께 운영할 때 공통 라이브러리는 과도하게 키우기 쉽습니다.  
공유 대상은 엄격하게 제한해야 유지보수 비용을 줄일 수 있습니다.

### 공유해도 되는 것

1. 프로토콜 스키마(Proto/OpenAPI/JSON Schema)
2. 인증 토큰 검증 규칙
3. 에러 코드 체계
4. 관측성 규약(필드명, trace 전파)

### 공유를 피해야 하는 것

1. 비즈니스 로직 전체
2. 언어별 구현 세부사항(ORM 모델, HTTP 클라이언트 구현)
3. 과도한 유틸 묶음(monolithic common lib)

### SDK 설계 원칙

1. 최소 API surface
2. semantic versioning 엄수
3. backward compatibility 우선
4. 오류 모델 표준화(`code`, `message`, `details`)

Java SDK와 Go SDK를 동시에 유지할 때는 "기능 동등성 테스트"를 자동화해 drift를 줄여야 합니다.

## 프로토콜/스키마 호환

언어가 달라도 계약은 하나여야 합니다.  
마이그레이션 실패의 대부분은 코드보다 계약 불일치에서 발생합니다.

### REST 호환 원칙

1. 필드 제거보다 deprecated -> 유예 -> 제거 단계 준수
2. 새로운 필드는 optional로 추가
3. enum 확장은 클라이언트 fallback 고려
4. 에러 응답 포맷 고정

### gRPC/Proto 호환 원칙

1. field number 재사용 금지
2. 삭제 시 `reserved` 사용
3. required 성격 필드는 점진 전환
4. 서버/클라이언트 버전 skew 허용 범위 정의

### 데이터 스키마 진화

DB/이벤트 스키마 모두 "expand -> migrate -> contract" 전략이 안전합니다.

1. expand: 새 컬럼/필드 추가(구버전 호환)
2. migrate: 양쪽 코드 동작 확인 후 데이터 이행
3. contract: 구버전 의존 제거 후 정리

### 계약 테스트

1. consumer-driven contract test
2. snapshot 기반 응답 검증
3. Java/Go 구현 간 golden test 비교

계약 테스트를 CI 게이트에 포함하면 릴리즈 시 호환성 사고를 크게 줄일 수 있습니다.

## 운영/관측성 일관성

Java 서비스와 Go 서비스를 함께 운영하는 기간에는 관측성 기준을 통일해야 비교/분석이 가능합니다.

### 공통 운영 규약

1. 공통 request_id/trace_id 전파
2. 공통 로그 필드(`service`, `route`, `status`, `latency_ms`)
3. 공통 메트릭 이름 규칙(RED + 리소스)
4. 공통 알람 임계치 체계

### 트래픽 전환 시 모니터링 포인트

1. 에러율 증가 여부
2. p95/p99 지연 시간 변화
3. 다운스트림(DB/Redis/MQ) 부하 변화
4. 비즈니스 KPI(주문 성공률 등)

### 장애 대응 플레이북

1. canary 단계별 자동 중단 조건
2. 즉시 rollback 커맨드/절차 문서화
3. 사후 분석(postmortem) 템플릿 통일

### 조직/프로세스 관점

1. Java/Go 리뷰어 페어링으로 지식 전파
2. 운영 온콜 핸드오프 기준 명확화
3. 마이그레이션 완료 기준(기능/성능/운영) 합의

코드 마이그레이션보다 중요한 것은 "운영 책임 모델"의 전환입니다.

## 요약

1. Go 마이그레이션은 전면 재작성(big-bang)보다 점진 분리 전략이 안전하다.
2. 공통 SDK/라이브러리는 계약 중심으로 최소화해야 한다.
3. REST/gRPC/데이터 스키마 호환 규칙을 선제적으로 고정해야 한다.
4. Java와 Go의 관측성/운영 규약을 맞춰야 전환 리스크를 줄일 수 있다.

## 체크리스트

- 분리 대상 기능이 기술/운영 지표 기준으로 선정되었는가
- 계약(OpenAPI/Proto/이벤트 스키마)이 CI에서 검증되는가
- canary/rollback 기준이 수치로 정의되어 있는가
- Java/Go 공통 로그/메트릭/트레이스 규약이 있는가
- 마이그레이션 완료 기준이 팀 간 합의되어 있는가
