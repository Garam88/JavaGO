# 01. Go 한눈에 보기: Java 개발자가 먼저 알아야 할 핵심

## Go 철학

Go는 "언어 기능을 늘려 복잡한 문제를 해결"하기보다, "단순한 도구를 조합해 시스템을 안정적으로 운영"하는 쪽에 가깝습니다.  
Java 개발자가 처음 체감하는 차이는 아래 4가지입니다.

1. 단순성: 문법 요소가 상대적으로 적고, 규칙이 일관적입니다.
2. 명시성: 숨은 동작보다 코드에 드러나는 동작을 선호합니다.
3. 빠른 컴파일: 개발 루프(수정 -> 빌드 -> 실행)가 짧습니다.
4. 표준 라이브러리 중심: 웹, JSON, 테스트 같은 핵심 기능을 기본 제공하는 편입니다.

Java와 비교해 감각을 잡으면 다음과 같습니다.

- Java: 프레임워크 중심으로 계층/패턴을 먼저 정하는 경우가 많음
- Go: 표준 라이브러리 + 작은 패키지 조합으로 문제를 점진적으로 해결

이 책도 같은 원칙을 따릅니다.

1. 먼저 표준 라이브러리로 동작하는 코드를 만든다.
2. 병목이나 반복이 명확해질 때 외부 라이브러리를 도입한다.
3. 추상화는 "미래 대비"보다 "현재 요구"를 기준으로 추가한다.

## 객체지향 대신의 사고방식

Go에는 Java의 `class`/`extends` 구조가 없습니다. 대신 아래 조합으로 설계합니다.

1. 타입(`struct`): 데이터를 표현
2. 메서드(receiver): 타입에 동작을 부여
3. 인터페이스(interface): 필요한 행위를 최소 단위로 정의
4. 임베딩(embedding): 조합으로 재사용

핵심은 "상속"보다 "조합 + 작은 인터페이스"입니다.

Java 스타일로 먼저 인터페이스를 크게 설계한 뒤 구현체를 맞추는 방식은 Go에서 과한 추상화가 되기 쉽습니다.  
Go에서는 보통 다음 순서를 권장합니다.

1. 구체 타입부터 만든다.
2. 실제로 대체가 필요한 지점에서 작은 인터페이스를 도입한다.

예시:

```go
type OrderRepository interface {
	Save(ctx context.Context, o Order) error
}
```

`OrderRepository`는 "저장" 행위 하나만 노출합니다.  
이렇게 인터페이스를 작게 유지하면 테스트 대체(fakes/stubs)와 구현 교체가 쉬워집니다.

또 하나 중요한 차이는 "예외"입니다.

- Java: `try-catch` 중심
- Go: `error` 값을 반환하고 호출자가 처리

즉, 제어 흐름을 예외에 맡기기보다, 상태를 명시적으로 전달하는 방식이 기본입니다.

## 요약

1. Go는 단순성/명시성/빠른 컴파일/표준 라이브러리 중심의 언어다.
2. 클래스 상속보다 `struct + method + interface` 조합이 핵심이다.
3. 인터페이스는 구현 이전이 아니라 "필요한 경계"에서 작게 도입한다.

## 번외

### Java 개발자가 초기에 자주 하는 실수

1. 프레임워크 없이도 가능한 문제를 너무 일찍 추상화함
2. 인터페이스를 구현 전에 크게 설계함
3. 예외처럼 `panic`을 일반 에러 처리에 사용함
4. 패키지 경계를 무시하고 순환 참조를 만들기 시작함

초기에는 "작은 패키지 + 명확한 의존 방향 + 테스트 가능한 함수"에 집중하면 됩니다.
